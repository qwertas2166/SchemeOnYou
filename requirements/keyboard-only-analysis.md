# Анализ keyboard-only workflow для разработчика

## Контекст

Целевая аудитория — разработчик, который хочет создавать диаграммы преимущественно с клавиатуры. Мышь разрешена, но не должна быть обязательной: приложение должно быть полноценно управляемым с клавиатуры.

## Почему разработчик может хотеть keyboard-only

1. **Скорость**
   - Не нужно переключаться между клавиатурой и мышью.
   - Быстрое создание элементов через команды.
   - Повторяемые действия можно выполнять сериями.

2. **Поток мышления**
   - UML часто рождается как структура: классы, методы, связи, сценарии.
   - Структуру быстрее описывать текстом/командами, чем перетаскивать фигуры.

3. **Привычки IDE**
   - Разработчики привыкли к IntelliJ/VS Code/Vim/Emacs: command palette, fuzzy search, shortcuts, navigation stack.
   - Ожидается UX уровня IDE, а не офисного редактора диаграмм.

4. **Доступность**
   - Keyboard-only повышает accessibility.
   - Полезно пользователям с ограничениями моторики или без точного pointing device.

5. **Автоматизация**
   - Текстовые команды проще записывать, повторять, генерировать.
   - Возможен future сценарий scripting/macros.

## Риски keyboard-first подхода

1. **Сложность визуального layout'а**
   - Расставлять элементы без мыши сложно.
   - Нужны auto-layout и команды относительного позиционирования.

2. **Высокий порог входа**
   - Пользователю нужно выучить команды и shortcuts.
   - Нужны discoverability: command palette, help, подсказки.

3. **Конфликт с ожиданиями GUI**
   - Пользователь может ожидать drag-and-drop.
   - Но цель проекта — именно keyboard-first; мышь может быть optional, не primary.

4. **Сложность редактирования связей**
   - Связи требуют выбора source/target/type.
   - Нужен быстрый fuzzy выбор элементов и smart defaults.

## UX-принципы

1. **Keyboard-first, not keyboard-afterthought**
   - Любое действие доступно с клавиатуры.
   - Мышь допустима как дополнительный способ управления.
   - Мышь не обязательна для выполнения полного сценария.

2. **Command palette как главный entry point**
   - `Ctrl+K` / `Ctrl+Shift+P`: открыть команды.
   - Команды ищутся fuzzy search.
   - Примеры: `Add participant`, `Add message`, `Add table`, `Add foreign key`, `Rename`, `Export`.

3. **Modal editing без навязывания Vim**
   - Можно поддержать режимы: navigation / insert / command.
   - Но базовый UX должен быть понятен без знания Vim.

4. **Структурная навигация**
   - Перемещение не только по координатам, но и по модели:
     - следующий класс
     - предыдущая связь
     - перейти к родителю
     - перейти к связанному элементу
     - найти элемент по имени

5. **Auto-layout by default**
   - При создании элементов layout строится автоматически.
   - Ручные layout-команды — вторичны.

6. **Текстовое редактирование свойств**
   - Выбранный элемент редактируется в панели свойств как текст.
   - Например class body, attributes, methods, stereotypes.

7. **Predictable hotkeys**
   - `Tab` / `Shift+Tab` — навигация по focus areas.
   - `Enter` — редактировать/подтвердить.
   - `Esc` — выйти/отменить.
   - `Delete` — удалить выбранное только после подтверждения или undo-safe.
   - `Ctrl+Z` / `Ctrl+Shift+Z` — undo/redo.
   - `Ctrl+S` — сохранить.
   - `Ctrl+F` — найти.

## Рекомендуемый основной сценарий

1. Пользователь создает диаграмму: `Ctrl+K` → `New sequence diagram` или `New database diagram`.
2. Для sequence diagram добавляет participant/message через command palette или короткие команды.
3. Для DB diagram добавляет table/column/foreign key через command palette или панель свойств.
4. Связи создаются через fuzzy выбор source/target и типа связи.
5. Приложение автоматически раскладывает диаграмму на canvas.
6. Пользователь навигирует по элементам через `Tab`, стрелки, поиск.
7. Экспортирует в SVG. Остальные форматы вынесены за рамки MVP.

## Вывод

Keyboard-first разработчику нужен не обычный canvas с hotkeys, а IDE-подобный редактор моделей:

- модель первична;
- диаграмма является визуальным представлением;
- команды и fuzzy navigation важнее палитры фигур;
- auto-layout обязателен;
- все операции должны быть undo-safe.
